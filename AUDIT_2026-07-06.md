# Code Audit — Restocker / Bank bot / CSN Sales mod
**Date:** 2026-07-06 · **Scope:** RestockerLocal (bot + web + db), Bank bot, Sales (CSN mod)

---

## 1. Executive summary

The project is a three-part Minecraft shop-economy system: the **CSN Fabric mod** (Sales) scrapes shop
transactions to CSV, **Restocker** (Discord bot + aiohttp web app + SQLite) is the source of truth for
wallets / stock / orders, and the **Bank bot** layers savings, loans, bonds and investing on top over a
token-authenticated HTTP API.

Overall the code is **well-structured and defensively written** for a hobby-scale project: money moves are
guarded, the bank API uses constant-time token comparison, web sessions use CSRF + secure cookies, and the
SQLite layers are clean. Static analysis found **no undefined-name crashes** anywhere.

The audit found **one critical correctness bug** (workers cannot complete claimed orders due to a
string/int type mismatch on `user_id`) plus three previously-fixed reliability bugs in the order-fulfil
flow, and a handful of medium/low issues and hygiene items. The critical bug and the three fulfil-flow
bugs have been fixed in this session; the remaining items are recommendations.

---

## 2. Architecture sketch (what each part does)

### Sales — CSN Export mod (Fabric, MC 1.21.11)
- `CsnExportClient.java`: press K → pages through `/csn history`, dedupes with a per-period `.seen`
  SHA-256 set, writes `csn_export_*.csv` + `csn_monthly_*.csv`, runs `/csn clear`, then **verifies** the
  clear and retries. Second keybind scans live shop stock → `csn_stock_*.csv`.
- **Verb semantics (critical):** `bought` = customer bought FROM you = your income; `sold` = your expense.
- Dedup invariant: monthly totals use the pre-update `.seen` set so a failed clear can't double-count.

### RestockerLocal
- `main.py`: launcher → `Restocker_main.py`.
- `Restocker_main.py` (~8.2k lines): bot engine — economy helpers (`add_coins`/`deduct_coins`,
  `apply_weekly_interest`), order lifecycle (`_apply_claim`, `update_order_messages`, DM panels),
  stock exchange exec, CSN parsing, loyalty/teams, and the `on_ready` persistent-view registration.
- `Restocker_db.py` (~1.9k lines): pure-SQLite store (no Discord) — balances, coin ledger, items,
  orders + order_claims, investors/ETF, markets, teams, projects, platform balance, config.
- `Restocker_web.py` (~2.4k lines): aiohttp site + JSON API; login codes, sessions, CSRF, market pages.
- `bank_api.py`: token-protected `/api/bank/*` routes mounted into the web app (balance/adjust/transfer/
  stock buy-sell/portfolio) — the boundary the Bank bot calls.
- `cogs/`: slash-command groups (admin, brew, config, devassist, events, inventory, loops, loyalty,
  market, misc, money, orders, projects, reports, shop, stock, team, tool).
- `views/`: Discord UI (orders, stock, web, hive).

### Bank bot
- `bank_main.py`: all `/bank`, `/loan`, `/bond`, `/invest` commands + a daily interest loop.
- `bank_db.py`: pure-SQLite store (accounts, savings, loans, bonds, ledger).
- `restocker_client.py`: async HTTP client for Restocker's `/api/bank/*`; every money move debits/credits
  the Restocker wallet first, then records locally, refunding on remote failure.

### Data flow
`CSN mod CSV → Restocker (wallet + stock + orders, restocker.db = truth) → Bank bot (bank.db, via HTTP API)`

## 3. Verification & testing performed

- **Static analysis (pyflakes)** across every Python file. No undefined names, no redefinition bugs.
  Only unused imports and two cosmetic "f-string without placeholder" warnings.
- **Dummy-data unit tests on `Restocker_db.py`** (temp SQLite): balances, ledger, items (upsert/rename/

- **Dummy-data unit tests on Bank bot `bank_db.py`**: accounts, savings (deposit/overdraft-guard),
  loans (payment, overpay-can't-go-negative, auto-close), bonds (create/claim-lock/redeem),
  ledger — **16/16 passed.** Idempotency guards (claim lock, overdraft rejection) all hold.
- **Focused reproduction** of the critical bug through the real save→load path (below).
- **Auth review**: bank API token (`hmac.compare_digest`, 503-when-unset, `@require_token` everywhere)
  and web sessions (one-time expiring login codes, `secrets` tokens, httponly/secure/SameSite cookies,
  per-session CSRF) — both sound.

## 4. Findings

### 🔴 CRITICAL — Claimants cannot complete their own orders (`user_id` string/int mismatch) — FIXED
`order_claims.user_id` is stored as **TEXT**, so orders reloaded from the DB (every handler does
`load_orders()`) return claims with a **string** `user_id`. Four ownership checks in `views/orders.py`
compare it directly to `interaction.user.id` (an **int**):

- line ~1053 `has_claim` in **fulfill** ("you must have a claim to submit proof")
- line ~1095 `has_claim` in **add produced**
- lines ~321 & ~863 in **release claim**

`"555…" == 555…` is always `False`, so the claimant is wrongly blocked. Reproduced end-to-end: a claim
saved with an int id reloads as a string and the check returns `False`. **This is the true root cause of
the "Fulfilled does nothing" report** (compounded by the ephemeral-in-DM bug that hid the message).
`get_claimers()`, `ManagerReviewView`, and `_is_blocked_claimer` already coerce types, so only these four
sites were affected.
**Fix applied:** coerce every claim's `user_id` to `int` once in the `load_orders()` wrapper in
`Restocker_main.py`, so all downstream comparisons work uniformly.

### 🟠 Order-fulfil reliability (three bugs) — FIXED earlier this session
1. **Ephemeral replies in DMs** — `_fulfill_core`'s `reply()` hardcoded `ephemeral=True`, which Discord
   rejects in a DM; the error was swallowed, so the worker saw nothing. Now drops ephemeral off-guild.
2. **`OrderView` lost its order id after a restart** — persistent view registered as `OrderView(0)`;
   buttons used static custom_ids. Added message-based `_oid()` recovery (race-safe).
3. **`ManagerReviewView` not persistent** — Approve/Reject buttons had no custom_ids and weren't
   registered, so they died on restart (managers couldn't approve → workers unpaid). Added custom_ids,
   registration, and interaction-based id recovery.

### 🟡 MEDIUM — Non-atomic balance updates (read-modify-write race)
`add_coins`/`deduct_coins` do `get_balance` → compute → `set_balance` as separate statements, and
`apply_weekly_interest` / `_save_balances` load the **whole** balances table, mutate in memory, and write
it back. Two concurrent coin operations (e.g. a payout during the interest loop) can clobber each other
and lose coins. Low probability at current scale, but it's real money.
**Recommendation:** do balance deltas in a single atomic SQL `UPDATE … SET coins = coins + :d` under the
existing connection, and drop the whole-table write path.

### 🟡 MEDIUM — Type consistency of user IDs across tables
The root cause of the critical bug is a broader pattern: some columns store IDs as TEXT, some code
compares as int. Pick one representation (recommend **int** everywhere for Discord snowflakes, or **str**
everywhere) and centralize coercion in the DB layer rather than at call sites.

### 🟢 LOW / hygiene
- Two f-strings without placeholders: `Restocker_web.py:2380`, `views/web.py:301` (cosmetic).
- Unused imports in `Restocker_main.py` (`hashlib`, `tasks`, `View/Button/Select`) and an unused local
  `io_bytes` (~line 5141).
- Web sessions never expire server-side (only the 30-day cookie); consider a stored expiry + rotation.
- Repo hygiene: duplicate `* - Copy.py` files, a committed 54 MB `cloudflared.exe`, and the live
  `restocker.db`/`-wal` in the working tree. Add these to `.gitignore` and remove the copies.

## 5. Recommendations to make the code better (senior-dev view)

1. **Centralize claim/ownership logic.** Add one helper (e.g. `user_holds_claim(order, user_id)`) that
   coerces types, and use it at all four sites plus `get_claimers`. Prevents this class of bug recurring.
2. **Atomic money mutations.** Single-statement SQL deltas for coins/principal; wrap multi-step economic
   actions (order approval payout, bank transfer) in one transaction so partial failures roll back.
3. **Persistent-view pattern.** The `OrderView`/`ManagerReviewView` restart bugs came from stateful
   persistent views. Standardize on `discord.ui.DynamicItem` with the id encoded in the custom_id, so
   state is recovered from the interaction by construction (no shared-instance workarounds).
4. **A real test suite.** The two dummy-data harnesses written here (DB layers) should become a `tests/`
   pytest package and run in CI. They already catch the critical bug. Add order-lifecycle and
   bank↔restocker consistency tests next.
5. **Reduce `Restocker_main.py` size.** At ~8.2k lines it's hard to reason about; the cog/view extraction
   is a good start — continue moving economy + CSN parsing into their own modules with unit tests.
6. **Structured logging + metrics** on money paths (every `add_coins`/`deduct_coins`/transfer already
   writes a ledger row — surface a periodic reconciliation check that Restocker wallet == bank's view).
7. **Repo hygiene** as above (gitignore binaries/db, delete `- Copy` files).

## 6. Testing limitations / honesty note

Full end-to-end testing requires a live Discord gateway + guild, which can't be stood up here, so the
Discord-facing command/interaction handlers were verified by **reading + static analysis + targeted
reproduction**, not by executing them against Discord. The **pure-logic layers (both SQLite DBs) were
executed with dummy data** and are the strongest-verified part. A full line-by-line reading of all ~21k
lines was not done; the review was risk-prioritized (money, orders, auth, persistence).

## 8. Manual follow-up you must do (git hygiene)

The live DB and the `* - Copy.py` files are still **tracked** in git even though `.gitignore` lists them —
they were committed before being ignored, which is what causes `git pull` to conflict. This couldn't be
fixed from the audit environment (the sandbox mount can't safely write to `.git`), and a stale
`.git/index.lock` was left behind. **In your own terminal, in the RestockerLocal folder, run:**

```
del .git\index.lock            REM delete the leftover lock first (Windows)
git rm --cached restocker.db restocker.db-wal restocker.db-shm
git rm "Restocker_db - Copy.py" "bank_api - Copy.py" "fix_catalog_stacks - Copy.py" "main - Copy.py"
git commit -m "Stop tracking live DB + remove duplicate Copy files"
```

(The Copy files remain recoverable from git history. No source files were deleted by the audit.)
